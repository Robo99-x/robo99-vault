### 2-1. watchlist .state 시드 (P1)

**문제:** watchlist.md에 `[[삼성전기]]`, `[[NVDA]]` 등 11개 등록됐는데 `tickers/.state/`에 매칭 파일 없음. 매일 premarket 브리핑이 "UNRESOLVED_ALIAS 15건" 경고.

**해결:** Python one-liner로 15개 .state YAML 생성 (삼성전기, 삼화콘덴서, LITE, NVDA, 씨엠티엑스, 한스바이오메드, 파마리서치, 삼천당제약, LS전선, 두산에너빌리티, CCJ, AT&T, 현대로템, LS ELECTRIC, GLW).

**검증:** `compile_premarket_context.py` 재실행 → `UNRESOLVED_ALIAS 15건 → 0건` ✅

### 2-2. theme_screener 0건일 때 Claude 스킵 (P1)

**문제:** `scheduler.log` 분석 결과, `job_theme_screener`가 0건일 때도 Claude를 호출하고 매번 600초 × 2 = 20분 타임아웃 발생.

**해결:** `scripts/scheduler_daemon.py` `job_theme_screener()`에 early exit 추가.
- stocks가 0건이면 즉시 return
- 추가로, 107개 `.state` 파일 전체 glob 대신 `last_seen` 있는 종목 이름만 파이썬에서 미리 추출해 문자열로 주입 (프롬프트 경량화)

### 2-3. 수급 특징주 섹션 (P1)

**문제:** 종목 .md에 "이 종목이 언제 왜 올랐는지" 히스토리가 안 쌓임. 기존 `## 특징주 기록` 섹션은 verbose (### 날짜 + 5줄 불릿).

**해결:** `scripts/entity_syncer.py` 변경
- 섹션명: `## 특징주 기록` → `## 수급 특징주`
- 포맷: `### date` + 불릿 → 한 줄 (`YY/MM/DD [테마] +X% 대금Xx 거래량Xx 태그`)
- 소스: 테마 스크리너만 (15:40 결과). stage2는 .md에 안 기록.
- 같은 날짜 중복 방지

**처음에 stage2도 .md에 기록하도록 만들었다가 형님이 "내가 원한 건 왜 올랐는지 narrative였지 수치 나열이 아니야"라고 피드백 → 되돌림.**

### 2-4. compile_channel_mentions.py 신규 (Karpathy 첫 단계)

**배경:** Karpathy의 "LLM Knowledge Bases" 글을 형님이 공유.
> raw 데이터를 LLM이 .md wiki로 compile, 모든 항목이 backlink로 연결.

**상황:** `40_consensus/raw/`에 메리츠/cahier_de_market 채널 66개 게시물 누적. 종목코드(6자리) + 한글 종목명 + 풍부한 narrative 포함. 하지만 종목 .md와 연결 안 되어 있음.

**설계 결정:**
- LLM 없이 결정적(deterministic) 매칭으로 시작 (코드 + substring)
- 종목 .md에 `## 채널 멘션` 새 섹션 추가, 다른 섹션 절대 안 건드림
- Idempotent (재실행해도 동일)

**구현:** `scripts/compile_channel_mentions.py`
- `tickers/.state/*.yaml`에서 name→ticker, code→name 맵 구축
- `40_consensus/raw/**/*.md` 전부 스캔
- 6자리 코드 regex + 3자 이상 한글명 substring 매칭
- 매칭된 종목 .md에 `## 채널 멘션` 섹션 갱신 (채널별 그룹, 날짜 내림차순, 최근 30건)

**결과:**
| 지표 | 값 |
|------|----|
| 처리 raw 파일 | 66개 |
| 매칭 종목 (≥1 멘션) | 13개 |
| 멘션 발견 파일 비율 | 27/66 (41%) |
| `.md` 갱신 완료 | 11개 |
| 스킵 (.md 없음) | 2개 (삼성전자, 코리아써키트) |

상위 멘션: 삼성전자(9), 덕산네오룩스(6), 삼성전기(6), DB하이텍(4), 이수페타시스(3)

**검증:**
- ✅ 기존 manual 섹션 (핵심 펀더멘털 등) 그대로 보존
- ✅ Markdown 링크 정상 (Obsidian에서 클릭 가능)
- ✅ Idempotent — 재실행 후 hash 동일

## 💡 배운 것 / 결정한 것

1. **Karpathy 방식 = 우리가 이미 만든 구조와 일치.** 4레이어가 정확히 raw→wiki→ops→schema에 대응. 빠진 건 LLM compiler 한 단계.
2. **결정적 매칭부터 시작.** LLM compiler는 비싸고 비결정적. 1차로 코드+이름 substring 매칭만 해도 27/66 파일에서 13종목 backlink 생성. 충분.
3. **데이터 분산 방지.** CLAUDE.md상 `20_wiki/tickers/`가 canonical이지만 실제론 `tickers/`가 활성. 데이터를 두 곳에 분산시키지 말고 일단 `tickers/`로 통일. 마이그레이션은 별도 태스크.
4. **수치 vs 내러티브.** 형님은 "왜 올랐는지" 텍스트를 원함. 단순 수치 누적이 아니라 의미 있는 narrative가 가치. 향후 LLM 기반 compile_overview.py에서 채널 멘션을 합성해 기업개요로 만들면 진짜 Karpathy.
